# AI 总览重构设计

## 目标

重构“总览”模块中的 AI 总览，让它能够：

1. 基于宠物档案、选定时间段内的待办、提醒和记录进行综合分析。
2. 输出高密度、可执行、按重要且紧急排序的建议，而不是泛泛而谈。
3. 降低 AI 生成时间，减少无效上下文和重复输入。
4. 在 UI 上突出“全局结论优先、按宠物深入分析”的阅读路径。

## 背景问题

当前 AI 总览链路已经具备基础能力，但仍有几个明显问题：

1. 输入给 AI 的上下文存在重复字段和较重的样本负担，容易拖慢生成。
2. 页面呈现更接近“结构化报告平铺”，不够贴近用户决策。
3. 全局判断、建议优先级和分宠物分析之间的层级还不够清晰。
4. “该做但未做”的判断还不够突出，缺少宠物特性和照护规则模板的辅助。

## 设计原则

1. 输入只提供事实证据，不提前塞入 AI 需要输出的结论。
2. 首屏只呈现全局信息，详细区只呈现单宠物信息。
3. 建议排行榜必须先满足宠物覆盖，再按重要且紧急排序。
4. 评分不是统计流水账，而是判断“该做的是否被安排、关注和闭环”。
5. 优先保留最影响判断的信号，删除非必要上下文，缩短生成时间。

## 产品结构

### 顶部控制区

顶部右侧区域放两个操作：

1. 左侧 `配置` 按钮
2. 右侧 `生成总览` 胶囊按钮

点击 `配置` 后弹出弹窗，用于统一设置：

1. 时间段：`7天 / 1个月 / 3个月 / 6个月 / 1年 / 自定义`
2. 宠物选择：支持多选，默认全选当前宠物

生成总览时，只分析当前配置内的时间段和宠物。

### 首屏结构

首屏只展示全局判断，不展示单宠物详细分析：

1. 全局总分大数字
2. 全局状态语
3. 一句话总结
4. 建议排行榜

### 建议排行榜

建议排行榜是首屏最重要的信息载体，规则如下：

1. 按“重要且紧急”排序
2. 条数下限为 `max(5, 已选宠物数)`
3. 每只已选宠物至少有 1 条建议
4. 如果宠物数不足 5，只要还有重要问题，就继续补足到 5 条
5. 多宠物时每条建议必须显式点名宠物
6. 建议可混合三类：
   - 风险提醒
   - 信息缺口
   - 下一步动作

### 详细分析区

详细分析区只按宠物分类展示，不再提供 `全部` 标签页。

1. 顶部显示宠物标签切换器
2. 每个标签页对应一只已选宠物
3. 默认选中建议排行榜第 1 条涉及的宠物中，在配置顺序里最靠前的那只
4. 每个宠物页展示：
   - 该宠物分数
   - 该宠物状态语
   - 五段式详细分析

详细分析分段统一为：

1. 为什么是这个分数
2. 当前最该处理什么
3. 你漏了什么
4. 最近有哪些变化
5. 后续怎么跟

## 评分逻辑

### 基本思路

评分不再只看“做了多少”，而是看“该做的有没有做、有没有跟、有没有闭环”。

AI 需要结合以下信息综合分析：

1. 宠物档案特性
2. 最近时间段内的记录、提醒、待办
3. 明确的关注点和既往问题
4. 内置的少量通用照护规则模板

### 扣分信号

1. 该关注未关注
2. 该定期检查未安排
3. 提醒未及时完成
4. 提醒逾期或跳过
5. 对重点问题缺少持续记录

### 加分信号

1. 最近有针对关注点的有效记录
2. 已安排对应提醒或待办
3. 已完成复查或护理闭环
4. 记录对关键问题的支撑较完整

### 通用照护规则模板

这次只内置少量高价值规则模板，不做完整规则引擎。

首批规则模板面向：

1. 幼宠：疫苗、驱虫、成长观察
2. 老年宠：定期检查、体重和活动变化关注
3. 既往医疗/备注风险：复查、症状持续跟进
4. 过敏或明确特性：对应观察和记录完整度

### 状态语映射

状态语由 AI 输出，但必须遵守固定分数档位，不允许任意发挥：

1. `90-100`：状态良好
2. `80-89`：基本稳定
3. `70-79`：需要关注
4. `60-69`：风险偏高
5. `0-59`：优先处理

## AI 输入设计

### 原则

输入 JSON 只提供事实，不提供 AI 应该输出的结果。

因此输入中不应包含：

1. 全局总分
2. 单宠物分数
3. 状态语

### 输入骨架

输入骨架由三部分组成：

1. `analysisConfig`
2. `scoringGuidelines`
3. `pets`

#### analysisConfig

描述当前分析配置：

1. 时间范围
2. 已选宠物 ID
3. 已选宠物数量

#### scoringGuidelines

提供 AI 必须遵守的评分与状态语口径：

1. 状态档位
2. 扣分/加分规则
3. 建议排行榜覆盖与排序要求

#### pets

每只宠物只提供最小必要事实：

1. `profile`
2. `expectedCare`
3. `evidence`

其中 `evidence` 只保留：

1. 待办状态统计
2. 提醒状态统计
3. 少量高价值 `recentSignals`
4. 明确的 `missingCare`

### 明确删除的输入

为了降低生成时间，以下内容不再默认喂给 AI：

1. 全量待办、提醒、记录原始列表
2. 重复镜像的 scorecard 数据
3. 冗长 `note`、`summary` 原文
4. 对排序和判断无价值的普通样本

## AI 提示词设计

### System Prompt 目标

System Prompt 负责约束 AI 输出行为：

1. 只能基于输入证据分析
2. 输出 JSON
3. 先输出首屏结果，再输出按宠物分类的详细分析
4. 建议排行榜按重要且紧急排序
5. 每只宠物至少有 1 条建议
6. 多宠物时每条建议必须点名宠物
7. 详细分析按固定五段式输出
8. 没有证据时必须明确说明信息不足

### User Prompt 目标

User Prompt 只保留简短任务说明和压缩后的 JSON 数据，避免大段废话。

## AI 输出设计

AI 输出需要分为全局层和单宠物层。

### 全局层

1. 全局总分
2. 全局状态语
3. 一句话总结
4. 建议排行榜

### 单宠物层

每只宠物输出：

1. 单宠物分数
2. 单宠物状态语
3. 五段式详细分析

## 回退与边界

1. 如果 AI 生成失败，继续回退到当前本地规则总结，不破坏现有兜底链路。
2. 如果某只宠物在当前时间段样本不足：
   - 建议排行榜可以出现该宠物的信息不足建议
   - 单宠物页明确展示样本不足
3. 如果配置弹窗未选择宠物：
   - 默认全选当前宠物
4. 自定义时间段沿用现有时间范围逻辑，不扩散到无关模块。
